# Pipeline Runner

## Overview

여러 에이전트를 연결하여 전체 콘텐츠 파이프라인을 실행하는 오케스트레이터. MVP에서는 리서치 에이전트 → 원본 아티클 → 유튜브 에이전트의 순차 흐름을 실행한다. 에이전트 간 데이터 전달과 전체 실행 흐름 관리를 담당.

---

## Requirements

- REQ-001: 파이프라인은 YAML 파일로 정의. 실행할 에이전트 목록과 순서를 선언
- REQ-002: 파이프라인 정의 구조:
  - `name` — 파이프라인 이름
  - `agents[]` — 에이전트 실행 순서. 각 항목은 agent spec 파일 경로
  - `connections` — 에이전트 간 데이터 전달 규칙 (어떤 에이전트의 출력이 다음의 입력으로)
- REQ-003: ���이전트를 정의된 순서대로 순차 실행. 앞 에이전트의 출력을 뒤 에이전트의 입력으로 전달
- REQ-004: 에이전트 간 전달 데이터는 article-schema를 따르는 원본 아티클 (리서치 → 유튜브의 경우)
- REQ-005: 전체 파이프라인 실행 로그 — 에이전트별 시작/종료, 전체 소요시간, 총 토큰 사용량
- REQ-006: 에이전트 실행 중 에러 발생 시 해당 지점에서 정지. 사용자에게 재시도/스킵/중단 선택지 제공
- REQ-007: 파이프라인 실행 결과를 출력 디렉토리에 구조화하여 저장:
  - `output/{pipeline-name}/{timestamp}/`
  - `article.yaml` — 원본 아티클
  - `youtube/` — 유튜브 에이전트 출력 (스크립트, 오디오, 썸네일 등)

## User Journeys

### 전체 파이프라인 실행
1. 사용자가 파이프라인 실행 명령 (CLI)
2. 런타임이 파이프라인 YAML 로드
3. 리서치 에이전트 실행 → 원본 아티클 생성
4. 원본 아티클을 유튜브 에이전트에 전달
5. 유튜브 에이전트 실행 → 콘텐츠 풀세트 생성
6. 출력 디렉토리에 결과 저장
7. 전체 실행 요약 표시 (소요시간, 생성된 파일 목록, 토큰 사용량)

### 중간 에러 후 재개
1. 유튜브 에이전트의 voiceover 스텝에서 TTS API 에러
2. 파이프라인 정지. 에러 내용 + 선택지 표시
3. 사용자가 재시도 선택 → 해당 스텝부터 재실행
4. 또는 스킵 선택 → voiceover 없이 나머지 스텝 진행

### 리서치만 실행하고 나중에 유튜브 실행
1. 사용자가 리서치 에이전트만 단독 실행
2. 원본 아티클 YAML 파일 생성/저장
3. 사용자가 아티클을 검토/수정
4. 나중에 유튜브 에이전트를 해당 아티클 파일을 입력으로 단독 실행

## Edge Cases

- 리서치 에이전트 출력이 article-schema validation 실패 → 유튜브 에이전트로 전달하지 않음. 리서치 에이전트 재실행 또는 사용자 수정
- 파이프라인에 에이전트가 1개만 있는 경우 → 유효. 단독 에이전트 실행과 동일
- 파이프라인 실행 중 사용자가 중단 → 현재까지 생성된 결과는 출력 디렉토리에 보존

## Dependencies

- article-schema (에이전트 간 데이터 전달 포맷)
- agent-spec (파이프라인이 참조하는 에이전트 정의)
- agent-runtime (개별 에이전트 실행)
- research-agent (파이프라인의 첫 에이전트)
- youtube-agent (파이프라인의 두 번째 에이전트)

## Depends On This

- None (최상위 실행 레이어)
